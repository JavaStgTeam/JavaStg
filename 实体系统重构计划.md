# 实体系统重构计划

## 1. 重构背景与目标

### 1.1 背景
当前项目的实体系统存在以下问题：
- 基础实体类 `Obj` 过于庞大，包含过多职责
- 接口 `IPlayer`、`IItem`、`IEnemy`、`IBullet` 未被使用
- 实体类之间存在代码重复
- 坐标系统和渲染逻辑耦合在实体类中
- 控制方法（如移动、旋转等）直接在实体类中实现

### 1.2 重构目标
- 优化实体类的结构和职责划分
- 提高代码的可维护性和可扩展性
- 减少代码重复
- 清理无用接口和方法
- 使代码更加符合设计模式和最佳实践
- 支持kotlin语言
- 

## 2. 重构方案

### 2.1 核心架构调整

#### 2.1.1 重构 `Obj` 类
- **职责分离**：将 `Obj` 类拆分为多个职责单一的类
   - `BaseEntity`（抽象类）：包含核心实体属性和方法
   - `CoordinateSystem`：处理坐标转换逻辑
   - `ControlManager`：封装移动、旋转等控制方法
   - `RenderManager`：处理渲染逻辑

- **控制方法优化**:提供更多默认方法，如移动、旋转等
   - `moveTo`：移动到指定坐标,实体
      - `moveTo(x: Float, y: Float, time: Float)`：移动到指定坐标,耗时time秒
      - `moveTo(entity: Obj, time: Float)`：移动到另一个实体,耗时time秒
   - `moveBy`：移动指定距离
      - `moveBy(dx: Float, dy: Float, time: Float)`：使当前坐标增加(dx,dy),耗时time秒
      - `moveBy(angle: Float, range: Float, time: Float)`：使当前坐标增加另一个实体的坐标,耗时time秒
   - `moveAt`：研轨道运动,需要在util.math中的轨道类(未实现)
      - `moveAt(orbit: Orbit, speed: Float)`：以轨道运动,速度为speed
   - `rotateTo`：旋转到指定角度
      - `rotateTo(angle: Float, time: Float)`：旋转到指定角度,耗时time秒,x轴正方向为0,顺时针为正
   - `accelerate`：加速移动
      - `accelerate(acceleration: Float, time: Float)`：以加速度acceleration加速,耗时time秒,acceleration为负数时减速,速度小于0时反向
   - `rotate`：旋转实体
      - `rotate(angle: Float, time: Float)`：旋转实体,耗时time秒,x轴正方向为0,顺时针为正

- **接口清理**：
  - 移除未使用的接口：`IPlayer`、`IItem`、`IEnemy`、`IBullet`
  - 保留并优化 `IRenderable` 接口

#### 2.1.2 实体层次结构优化
- **建立清晰的继承体系**：
  - `BaseEntity`（抽象基类）
   - `Player`（玩家）
   - `Enemy`（敌人）
   - `Bullet`（子弹）
   - `Item`（物品）
   - `Laser`（激光）

### 2.2 具体实现计划

#### 2.2.1 第一阶段：基础架构重构
1. **创建核心类**：
   - `BaseEntity`（抽象类）：提取原 `Obj` 类的核心属性和方法
   - `ControlManager`：封装移动、旋转等控制方法
   - `CoordinateSystem`：将坐标转换逻辑移至此类
   - `RenderManager`：处理渲染逻辑

2. **清理无用接口**：
   - 移除 `IPlayer`、`IItem`、`IEnemy`、`IBullet` 等未使用的接口

3. **优化 `Obj` 类**：
   - 将 `Obj` 及其子类改为抽象类
   - 移除重复方法和冗余代码
   - 重构 `Obj` 类的子类
      - `Player`（玩家）
         - 不再继承字`Obj`, 而是直接做父类
      - `Enemy`（敌人）
         - 改为抽象类, 将默认敌人素材封装为属性
      - `Bullet`（子弹）
         - 目前不可用,需完全重构
      - `Item`（物品）
         - 目前不可用,需完全重构
      - `Laser`（激光）
         - 目前不可用,需完全重构

#### 2.2.3 第三阶段：系统集成与测试
1. **整合实体管理器**：
   - 优化 `GameWorld` 中的实体管理
   - 确保实体系统与其他系统的兼容性

2. **测试与验证**：
   - 验证重构后的功能完整性
   - 确保所有实体正常工作

3. **文档更新**：
   - 更新项目文档，包括类图、序列图和使用说明
   - 确保所有开发人员都了解新的架构和实现

4. **对象池使用程度优化**：
   - 分析当前对象池的使用情况
   - 确定是否需要增加或减少对象池的大小
   - 优化对象池的使用，避免内存泄漏和性能问题
   - 出入池时添加必要的逻辑, 便于统计obj数目
   
## 3. 预期成果

### 3.1 代码质量提升
- 职责清晰，代码结构合理
- 减少代码重复，提高代码复用性
- 提高代码可读性和可维护性
- 清理无用接口和方法

### 3.2 架构改善
- 建立清晰的实体层次结构
- 实现控制方法的封装
- 坐标系统逻辑独立
- 实体类改为抽象类，规范继承体系

### 3.3 可扩展性增强
- 更容易添加新的实体类型
- 更容易修改现有实体的行为
- 支持更多的游戏特性

## 4. 风险评估与应对策略

### 4.1 风险评估
- **功能回归**：重构可能导致现有功能失效

### 4.2 应对策略
- **渐进式重构**：分阶段进行重构，每个阶段都进行测试
- **充分测试**：在重构过程中不断进行测试，确保功能完整性
- **代码审查**：进行代码审查，确保代码质量
